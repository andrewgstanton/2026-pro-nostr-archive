# 

## Summary

This guide walks through installing modern PGP tooling on macOS using **GPGTools**, importing someone else's public key, generating your own keypair, and encrypting messages.  

It also explains how GPG resolves keys using email addresses and how to inspect key fingerprints.

This workflow remains one of the most durable methods for **cryptographic email encryption and identity verification**.

---

## Why Use PGP?

PGP (Pretty Good Privacy) provides:

- End-to-end encrypted communication
- Cryptographic identity verification
- Message signing
- Secure file encryption

A typical encryption flow looks like this:

```
You encrypt → with recipient's public key
Recipient decrypts → with their private key
```

Only the recipient can read the message.

---

# Step 1 — Install GPGTools on macOS

The easiest way to install modern OpenPGP tools on macOS is:

https://gpgtools.org

This installs:

- **GPG Suite**
- `gpg` command-line tool
- Keychain management UI
- Optional Apple Mail integration

After installation, confirm GPG works:

```bash
gpg --version
```

---

# Step 2 — Import Someone's Public Key

If someone sends you an ASCII-armored key (`.asc` file), import it like this:

```bash
gpg --import publickey.asc
```

Example result:

```
gpg: key E05D5901: public key "rhsafe@proton.me" imported
```

The key is now stored in your **local keyring**.

---

# Step 3 — List Keys in Your Keyring

To see all stored keys:

```bash
gpg --list-keys
```

Example output:

```
pub   ed25519 2022-04-14 [SC]
      E05D59019C6E9E3122A6C0A9A4FAF7C1F6F1AD7A
uid           rhsafe@proton.me <rhsafe@proton.me>
sub   cv25519 2022-04-14 [E]
```

Explanation:

| Field | Meaning |
|------|------|
| pub | primary key |
| ed25519 | signing algorithm |
| SC | signing + certification |
| sub | encryption subkey |
| cv25519 | encryption algorithm |

Modern PGP best practice is:

```
ed25519 (signing)
cv25519 (encryption)
```

---

# Step 4 — Generate Your Own Keypair

Create your own PGP identity:

```bash
gpg --full-generate-key
```

Typical prompts:

1. Select key type
2. Choose key length
3. Set expiration
4. Enter name + email
5. Create passphrase

Example result:

```
pub   rsa4096 2026-03-10 [SC] [expires: 2030-03-10]
      D5F7725DF6FBE40932EBE7F644861F7A61F5C1BB
uid           Andrew Stanton <andrewgstanton@gmail.com>
sub   rsa4096 2026-03-10 [E]
```

You now have:

```
public key   → share with others
private key  → keep secret
```

---

# Step 5 — Export Your Public Key

To allow others to send encrypted messages to you:

```bash
gpg --armor --export your@email.com
```

Output will look like:

```
-----BEGIN PGP PUBLIC KEY BLOCK-----
...
-----END PGP PUBLIC KEY BLOCK-----
```

You can:

- attach it to email
- publish it on a website
- upload to a keyserver

---

# Step 6 — Encrypt a Message

Create a message file:

```bash
nano message.txt
```

Example content:

```
Hello Roland,

Testing encrypted communication.
```

Encrypt it:

```bash
gpg --encrypt --armor --recipient rhsafe@proton.me message.txt
```

Output:

```
message.txt.asc
```

Only the owner of that private key can decrypt it.

---

# Step 7 — Decrypt a Message

If someone encrypts a message using **your public key**, decrypt it with:

```bash
gpg --decrypt message.asc
```

GPG will use your **private key** automatically.

---

# Step 8 — Understanding Key Identifiers

GPG can identify keys using several methods:

| Identifier | Example |
|------|------|
| email | rhsafe@proton.me |
| key ID | E05D5901 |
| fingerprint | E05D59019C6E9E3122A6C0A9A4FAF7C1F6F1AD7A |

The **fingerprint** is the most reliable.

---

# Step 9 — Viewing a Key Fingerprint

To inspect a key:

```bash
gpg --fingerprint rhsafe@proton.me
```

Example:

```
pub   ed25519 2022-04-14 [SC]
      E05D 5901 9C6E 9E31 22A6  C0A9 A4FA F7C1 F6F1 AD7A
```

Fingerprints are used to verify identities during secure exchanges.

---

# Notes on Trust

You may see this:

```
uid [ unknown ]
```

This simply means:

- you haven't verified the key
- you haven't signed it in your keyring

For normal encrypted communication this is fine.

## Choosing Modern OpenPGP Key Types (ed25519 + cv25519)

If you are generating a new PGP key today, the recommended configuration is:

```
Primary key:  ed25519   (signing / certification)
Subkey:       cv25519   (encryption)
```

This configuration is considered modern best practice and is widely used by tools such as Proton Mail.

### Why These Algorithms?

Elliptic curve cryptography provides strong security with much smaller keys and faster operations compared to traditional RSA keys.

| Algorithm | Typical Key Size | Notes |
|------|------|------|
| RSA 4096 | 4096 bits | older but still secure |
| ed25519 | 256 bits | modern elliptic curve signing |
| cv25519 | 256 bits | modern elliptic curve encryption |

Advantages of using `ed25519` and `cv25519`:

- smaller public keys
- faster signing and encryption
- modern cryptographic design
- widely supported by modern OpenPGP implementations

### Example Modern Key Layout

A modern OpenPGP key will typically look like:

```
pub   ed25519 2026-03-10 [SC]
uid           Your Name <your@email.com>
sub   cv25519 2026-03-10 [E]
```

Where:

- `ed25519` is used for signing and identity certification
- `cv25519` is used for encryption

### Compatibility

Modern OpenPGP implementations such as:

- GnuPG (GPG)
- Proton Mail
- GPGTools
- OpenKeychain

fully support these curves.

Older RSA keys are still secure and usable, but elliptic curve keys are generally recommended when generating a **new identity**.

### Regenerating a Key (Optional)

If you recently created an RSA key and have not shared it yet, you may choose to regenerate using elliptic curves.

Delete existing keys:

```
gpg --delete-secret-keys <key-id>
gpg --delete-keys <key-id>
```

Then generate a new key:

```
gpg --full-generate-key
```

Select:

```
ECC (sign and encrypt)
Curve: ed25519
```

GPG will automatically create the appropriate encryption subkey (`cv25519`).

### Relationship to Other Modern Systems

Many modern secure systems also use elliptic curves:

| System | Algorithm |
|------|------|
| Proton Mail | ed25519 / cv25519 |
| Signal | Curve25519 |
| WireGuard | Curve25519 |
| OpenSSH | ed25519 |
| Nostr | secp256k1 |

While the curves differ, they share the same principle: **small keys with strong security based on elliptic curve cryptography**.

## Deleting an Existing Key (Optional Reset)

If you recently created a key and have **not shared it with anyone yet**, you may choose to delete it and regenerate a new one (for example, if you want to switch to modern elliptic curve keys).

There are two ways to delete keys: using the **GPG Keychain application** (GUI) or the **command line**.

---

### Method 1 — Using GPG Keychain (macOS GUI)

1. Open **GPG Keychain** (installed with GPGTools).

2. Locate your key in the key list.

Example:

```
Andrew Stanton <your@email.com>
```

3. Right-click the key.

4. Select:

```
Delete Key
```

5. When prompted, choose to delete:

```
Public Key
Private Key
```

Once both are deleted, the key will disappear from your keyring.

---

### Method 2 — Using the Terminal

First list your secret keys:

```
gpg --list-secret-keys
```

Example output:

```
sec   rsa4096 2026-03-10
      D5F7725DF6FBE40932EBE7F644861F7A61F5C1BB
uid   Andrew Stanton <andrewgstanton@gmail.com>
```

Delete the **secret key first**:

```
gpg --delete-secret-keys D5F7725DF6FBE409
```

Then delete the **public key**:

```
gpg --delete-keys D5F7725DF6FBE409
```

Note: GPG requires deleting the secret key before removing the public key.

---

### Verify the Key Was Removed

Run:

```
gpg --list-keys
```

Your deleted key should no longer appear.

Any other imported public keys (for example keys belonging to other people) will remain in your keyring.

---

### Generate a New Modern Key

Once the old key is removed, generate a new one:

```
gpg --full-generate-key
```

Select:

```
ECC (sign and encrypt)
Curve: ed25519
```

GPG will automatically create:

```
Primary key: ed25519
Encryption subkey: cv25519
```

This configuration is widely used in modern OpenPGP environments.


## Generating a Modern OpenPGP Key (ed25519 + cv25519)

When generating a new PGP key using GnuPG on macOS (via GPGTools), you may notice that the default key generation menu only shows older key types such as RSA or DSA.

Example prompt:

```
Please select what kind of key you want:
   (1) RSA and RSA (default)
   (2) DSA and Elgamal
   (3) DSA (sign only)
   (4) RSA (sign only)
```

Modern elliptic curve options are **hidden unless expert mode is enabled**.

To generate a modern key, run:

```
gpg --full-generate-key --expert
```

This reveals additional key types.

Example menu:

```
Please select what kind of key you want:
   (1) RSA and RSA (default)
   (2) DSA and Elgamal
   (3) DSA (sign only)
   (4) RSA (sign only)
   (7) DSA (set your own capabilities)
   (8) RSA (set your own capabilities)
   (9) ECC and ECC
  (10) ECC (sign only)
  (11) ECC (set your own capabilities)
```

---

### Select the Key Type

Choose:

```
9
```

Which corresponds to:

```
ECC and ECC
```

This instructs GnuPG to create:

- an elliptic-curve primary key
- an elliptic-curve encryption subkey

---

### Select the Curve

When prompted for the curve, select:

```
Curve 25519
```

GnuPG will automatically configure the key as:

```
Primary key: ed25519  (signing / certification)
Subkey:      cv25519  (encryption)
```

---

### Verify the Key

After generation, confirm the result:

```
gpg --list-keys
```

Expected output:

```
pub   ed25519 2026-03-10 [SC]
uid           Your Name <your@email.com>
sub   cv25519 2026-03-10 [E]
```

Where:

| Component | Purpose |
|------|------|
| ed25519 | signing and identity certification |
| cv25519 | encryption |

Note that **cv25519 will not appear during key creation**.  
It only appears when viewing the final key with `gpg --list-keys`.

---

### Why This Matters

Elliptic curve keys are recommended for new OpenPGP identities because they provide:

- strong security
- smaller keys
- faster cryptographic operations
- broad support in modern tools such as GnuPG, Proton Mail, and GPGTools.
