## Afternoon Development Session

The remainder of the day was focused almost entirely on the publishing scheduler inside Continuum.

The scheduler originally supported notes, but the goal today was to extend that capability so that long-form articles could follow the same workflow.

By the end of the day, scheduled publishing for articles was functioning correctly.

The workflow now works like this:

1. Write the article locally.
2. Sign the event locally.
3. Choose a scheduled publish time.
4. Continuum stores the signed event and scheduling metadata.
5. When the scheduled time arrives, Continuum publishes the event to relays.

Because the event is already signed, the scheduler does not require access to the private key. It simply retrieves the signed event and sends it to the relay manager.

This maintains one of the key design principles behind Continuum: signing and publishing remain separate operations.

Signing happens locally with the identity.  
Publishing happens later through the network.

## Confirming Offline Behavior

Another important behavior was confirmed while testing the scheduler.

If the machine is offline when the scheduled publish time occurs, the event does not fail. Instead, it remains in the scheduled state.

When network connectivity returns and Continuum restarts, the scheduler checks the scheduled events table and immediately publishes any events whose scheduled publish time has already passed.

This means scheduled publishing is resilient to temporary network outages.

Offline conditions simply delay publishing rather than breaking it.

For a local-first publishing system, this behavior is extremely important.

## Article Scheduler Integration

Extending the scheduler to articles required a few small but important changes.

Notes use Nostr event kind `1`, while articles use kind `30023`. The scheduler therefore needed to support multiple event kinds.

The scheduler table already stores the following metadata:

- event id  
- event kind  
- scheduled publish timestamp  
- status

By recording the event kind alongside the event id, the scheduler can publish both notes and articles using the same logic.

This allowed the scheduling system to remain simple while supporting multiple types of events.

## Updating the Article API

After scheduling for articles was working internally, the article list page still needed to display scheduled status correctly.

Originally the `/api/articles` route only returned article records from the database and had no knowledge of the scheduler.

To fix this, the route was updated to join the articles table with the `scheduled_events` table.

If an article has a scheduled entry, the API now returns additional fields including:

- `scheduled_publish_at`
- `scheduled_publish_at_local`

These fields allow the frontend to display scheduling information without needing to know how the scheduler database is structured.

---

## UI Improvements

Once the API returned scheduling metadata, the article list view could finally display the correct state.

The interface now supports three article states:

Unpublished  
Scheduled  
Published

If an article is scheduled, the UI displays:

- a “Scheduled” badge  
- the scheduled publish timestamp  
- a disabled “Publish (Scheduled)” button  

This behavior now mirrors how scheduled notes already appear in the interface.

Keeping notes and articles consistent reduces confusion and makes the publishing workflow easier to understand.

---

## Timezone Handling

Scheduled timestamps are currently converted using a fixed timezone setting inside the API.

This works well enough for development and testing, but the long-term plan is to derive timezone information from the active identity configuration.

Continuum identities already store configuration data locally, so timezone support can eventually become identity-specific.

For now, the focus remained on validating the scheduler itself.

## End of Day

By the end of the afternoon session, scheduled publishing for articles was working and integrated across the backend API and the UI.

The system now supports:

- scheduling notes  
- scheduling articles  
- publishing scheduled events automatically  
- recovering from offline conditions  
- displaying scheduled status in the interface

These improvements strengthen Continuum’s role as a local-first publishing engine rather than simply a Nostr client.

Days like this often involve a series of small architectural improvements rather than a single visible feature. But each step tightens the system and moves it closer to a fully sovereign publishing workflow.
