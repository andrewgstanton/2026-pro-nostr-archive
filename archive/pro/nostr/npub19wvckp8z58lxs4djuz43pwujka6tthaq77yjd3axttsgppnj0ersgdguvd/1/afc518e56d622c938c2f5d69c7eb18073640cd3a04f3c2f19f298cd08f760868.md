More Improvements to Continuum’s Archive Queue System (5/23/2026)

Made several improvements tonight to the new queue-driven archive system in Continuum.

One important addition was handling events discovered from *other* Nostr clients correctly.

Previously, if I created a note or reply in another client like Primal and later refreshed data in Continuum, the event would appear locally in the database and dashboard, but it would never enter the archive queue because Continuum itself did not originally create it.

That is now fixed.

During refresh/import operations, newly discovered kind:0, kind:1, and kind:30023 events can now automatically be queued for archival if they are not already present locally or already queued.

This preserves the benefits of the original database-scanning archive model while still keeping the newer queue-driven persistence architecture.

I also split the archive sync process into two separate background tasks:

1. Commit Pending Events to Archive
2. Rebuild Global Index + RSS

The first task commits queued events into the Git-backed archive repositories.

The second task rebuilds the global archive index entirely from committed history files.

The archive queue page now displays active archive job information directly in the UI, including:

- current running archive task
- active thread names
- thread count
- runtime status information

This makes the archive subsystem feel much more transparent and understandable while background jobs are running.

One particularly satisfying moment tonight was being able to launch a long-running global archive index rebuild in the background while continuing to use Continuum normally for other tasks.

That separation between foreground authoring and background archival processing is starting to make Continuum feel much more like a durable local platform than a traditional web application.
