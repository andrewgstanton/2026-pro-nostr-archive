## Afternoon — Hardening Continuum for Offline Conditions

The majority of the afternoon was spent improving how Continuum behaves when relays or network connectivity are unavailable.

Nostr relays are often unreliable, and local-first systems must gracefully handle those conditions.

Rather than allowing users to attempt actions that will inevitably fail, I implemented UI-level safeguards that detect offline conditions and disable publish-related operations.

### Dashboard Changes

When the system detects that relays are unavailable, the dashboard now disables relay-dependent actions.

Specifically:

- DM buttons are disabled
- Relay interaction controls are disabled
- Data refresh controls are disabled

This prevents unnecessary errors and makes the system behavior clearer to the user.

### Notes Editor Changes

The same approach was applied across the notes publishing flows.

In the following screens:

- **Edit Note**
- **Edit Published Note**

Publish-related controls are now disabled when the system detects an offline state.

This includes:

- **Sign & Publish**
- **Sign & Republish**

These actions require relay connectivity, so disabling them avoids confusing failure states while still allowing local editing.

The goal is simple: users should always understand which actions are possible locally and which require network connectivity.

## Infrastructure Idea — Running a Personal Relay

Another idea that surfaced today was hosting a personal relay.

Many of the relays currently used are unreliable, which complicates publishing and synchronization.

One possible solution is to start with a **containerized relay** running locally or on a small server.

An initial configuration could:

- allow reads and writes only from approved pubkeys
- prevent spam
- provide reliable publishing for identities managed in Continuum

This would give much more deterministic behavior compared to relying entirely on public relays.
