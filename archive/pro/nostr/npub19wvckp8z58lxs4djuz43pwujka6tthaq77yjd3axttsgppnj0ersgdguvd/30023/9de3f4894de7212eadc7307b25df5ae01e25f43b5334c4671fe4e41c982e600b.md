## Continuum Development Progress

One of the most meaningful technical developments today was designing the architecture for **scheduled publishing** inside Continuum.

The key constraint I continue to enforce is simple:

**Continuum never stores unsigned events in the database.**

Drafts remain drafts. They exist only as local working content.

Because of this rule, scheduling must work differently from most publishing systems.

Instead of scheduling drafts, Continuum schedules **already signed events**.

This leads to a clean publishing flow:

- **Sign Only**
- **Sign & Publish**
- **Sign & Schedule**

When an event is scheduled, the system simply stores a minimal record that includes:

- the event ID
- the event kind
- the scheduled publish time

All actual content already exists in the appropriate event tables (articles or notes).

The scheduler then runs in the background and calls the existing publish mechanism:

`publish_event()`

This approach keeps the architecture simple, avoids duplication, and stays aligned with Continuum’s core philosophy:

**events are sovereign artifacts once they are signed.**

Scheduling is merely a delayed broadcast.

## Relay Runtime Improvements

I also continued refining the **relay runtime system**.

Relays now load dynamically when an identity is selected, and the relay manager maintains the active connection pool.

A relay status page now shows:

- the active identity
- the identity’s npub
- the currently active relays
- connection status

One improvement I plan to add is a timestamp showing when the relay set was last refreshed. This will make debugging relay behavior much easier.

These changes are gradually turning Continuum into something more than a publishing tool — it is becoming a **local Nostr operating environment**.
